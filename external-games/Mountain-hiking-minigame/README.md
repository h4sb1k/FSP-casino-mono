# Mountain-hiking-minigame

Фронтенд мини-игры про вертикальное восхождение (комнаты, ставки, бусты, события птица/лавина, лобби, приватные комнаты, подбор общей комнаты).

Текущая реализация является **игровым клиентом + локальной симуляцией** и предназначена как база для интеграции с сервером. Ниже описано:

- что уже работает на фронте;
- что должно быть перенесено/зафиксировано на бэкенде;
- как связать фронт и сервер;
- как подготовить репозиторий и загрузить на GitHub.

---

- React 18 + Vite 5
- CSS (ручная стилизация, анимации)
- Локальная симуляция раунда в `src/game/simulate.js`
- Структура:
  - `src/App.jsx` — UI, лобби/комната, таймеры, модалки, оркестрация раунда
  - `src/game/mockRound.js` — конфиг бустов, генератор плана раунда (mock)

  - `API_CONTRACT.md` — контракт интеграции (база для бэкенда)

## 2) Что реализовано на фронте (текущее поведение)

## 2.1 Лобби

- Список **общих** комнат (приватные скрыты из общего списка).
- Фильтры по:
  - входу (от/до),
  - количеству мест (от/до),
  - статусу.
- Кнопка `Подобрать комнату`:
  - выбирает первую доступную общую комнату со статусом `waiting` и свободными местами.
- Кнопка `Создать комнату` открывает модальный конфигуратор:
  - название комнаты;
  - вход;
  - мест;
  - ставка от/до;
  - флаг `Частная комната`.
- Если комната приватная, пользователь сразу заходит в неё после создания.

## 2.2 Комната

- Правая колонка:
  - блок участия (ввод ставки и подтверждение);
  - краткий статус ожидания: `Ожидание: ещё N участников для старта`;
  - кнопка `Выйти в лобби`;
  - блок бустов с модалкой подтверждения;
  - список участников;
  - лента событий;
  - панель победителя в конце раунда.
- Кнопка ставки:
  - активна/неактивна по фазе и валидности ввода;
  - при повторе той же суммы у подтверждённой ставки — `Ставка подтверждена`.

## 2.3 Участники и лидерборд

- Игрок отображается в комнате даже без ставки.
- Если игрок не подтвердил ставку, в таблице высоты показывается: `не участвует`.
- Если ставка подтверждена:
  - до старта: `0 м`;
  - в раунде/после: фактическая высота.
- На бегуне показывается бейдж с именем (`ВЫ` для локального игрока).

## 2.4 Ставки

- У комнаты есть диапазон `minBet..maxBet`.
- Подтверждение ставки:
  - списывает разницу из баланса (если ставка изменена);
  - сохраняет `roundBet` и `confirmed`.
- Пул раунда = сумма подтверждённых ставок участников.
- После завершения раунда пул начисляется победителю.
- После раунда все ставки и подтверждения сбрасываются.

## 2.5 Бусты (клиентская реализация)

- Доступны бусты: `helmet`, `boots`, `shield`, `lucky`.
- По нажатию открывается модалка с описанием.
- Кнопки модалки:
  - `Закрыть`
  - `Применить` (или `Отменить`, если этот буст уже выбран)
- Логика стоимости:
  - при смене буста стоимость старого возвращается;
  - списывается стоимость нового (фактически дельта);
  - при отмене выбранного буста стоимость возвращается.

## 2.6 Раунд и фазы

- Фазы: `waiting -> locked -> running -> finished -> waiting`.
- Старт раунда при минимум 2 подтверждённых участниках (без обязательного участия локального игрока).
- В раунде:
  - вертикальное восхождение;
  - птица (таргет-событие);
  - лавина (накатывающая снежная завеса + массовое воздействие).
- Анимации:
  - движение бегунов;
  - предупреждение перед лавиной;
  - попадания;
  - визуальный эффект победы/поражения.
- Победитель определяется по максимальному `progress` на финише.

## 2.7 Боты (только mock)

- В `waiting` автоматически добавляются боты до заполнения.
- Боты случайно подтверждают ставки в диапазоне комнаты.

## 3) Что сейчас на фронте, но должно быть на бэкенде

Ниже критично важные зоны, которые в production должны быть сервер-авторитативными:

1. **Создание/хранение комнат**
   - ID комнаты, тип (общая/приватная), конфиг, состав участников, статусы.

2. **Участие в комнате**
   - вход/выход игроков;
   - контроль лимита мест;
   - приватность и доступ.

3. **Баланс и ставки**
   - проверка средств;
   - списание/возврат/начисление;
   - защита от двойных списаний и гонок.

4. **Подтверждение участия в раунде**
   - кто реально участвует;
   - серверный снэпшот участников на старт.

5. **Бусты и экономика бустов**
   - выбор буста;
   - стоимость и возвраты;
   - модификаторы в расчёте раунда.

6. **План раунда и RNG**
   - генерация событий (`bird/avalanche`) только на сервере;
   - таргеты, шансы, резисты, штрафы.

7. **Физика/результат раунда**
   - обновление прогресса;
   - эффекты попаданий;
   - итоговая высота;
   - победитель.

8. **Таймеры фаз**
   - `waiting/locked/running/finished` с серверным временем.

9. **Лента событий**
   - события должны приходить из бэкенда (WS/SSE), а не собираться из локальной симуляции.

10. **Боты**
    - если нужны в проде, их поведение должно быть только серверным.

11. **Античит/консистентность**
    - клиент должен быть рендерером, а не источником истины.

## 4) Как соединить фронт с сервером (рекомендуемая схема)

Базируйтесь на `API_CONTRACT.md`.

## 4.1 Принцип

- Сервер: authoritative source.
- Клиент:
  - отправляет действия игрока;
  - получает состояние/события;
  - интерполирует и анимирует.

## 4.2 Минимальный транспорт

- HTTP:
  - лобби/комнаты;
  - создать комнату;
  - вход/выход;
  - ставка;
  - выбор/отмена буста;
  - старт раунда (или готовность к раунду).
- WS/SSE:
  - фазы раунда;
  - warning/event/finish;
  - изменения участников комнаты.

## 4.3 Синхронизация времени

- Все события содержат `serverTimeMs`.
- Клиент вычисляет offset и запускает визуал строго по серверным таймстемпам.

## 4.4 Что заменить в текущем фронте при интеграции

1. `createRoundPlan` -> убрать из боевого контура, заменить событиями сервера.
2. `makePlayers/tickPlayers/resolveEvent` -> оставить только как fallback mock-режим или удалить.
3. `addBotIfNeeded/confirmRandomBotBet` -> убрать в production (или перенести серверно).
4. `balance` и все списания/возвраты в `App.jsx` -> заменяются ответами backend.
5. `finishRound` должен принимать итог от backend (`winnerId`, `finalHeights`, `rewardDelta`).
